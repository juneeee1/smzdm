# CLAUDE.md - HTML文游通用开发规范

> 此文件对所有子项目自动生效，子项目的 CLAUDE.md 只需写项目特有内容。

## 1. 开发者画像与沟通协议
*   **身份**：非专业开发者，专注于**组件规范、用户体验、UI样式、交互**。
*   **沟通要求**：
    *   **先计划后代码**：修改前必须提供易懂的计划并由我确认。
    *   **增量修改**：采用小幅修改，解释代码作用，严禁大面积重写。
    *   **用中文沟通**：所有解释、计划、注释都用中文。

## 2. 代码组织与架构规范
*   **组件化开发**：功能拆分为独立文件，禁止将所有代码堆在单一文件中。
    *   每个功能模块一个 JS 文件，保持清晰的职责划分。
    *   若新功能导致文件过大，**必须进行拆分**。
*   **文件限制**：单个文件代码量必须控制在 **800 行以下**。
*   **详细规则**：编辑 HTML/CSS/JS 文件时，自动应用 `.claude/rules/` 中的对应规则。

## 3. 技术约束
*   **语言与库**：**仅限原生 JavaScript**（可选引入 jQuery）。**严禁**使用除 jQuery 外的任何第三方库。
*   **网络请求**：仅允许调用 **AI API**（用于角色对话、内容生成）。禁止引入其他外部服务依赖。
*   **实现方式**：
    *   **页面结构**：通过 JS 动态创建 DOM，HTML 文件只保留最小骨架。
    *   **样式**：CSS 文件与 JS 动态注入均可，保持一致性即可。
*   **适配性**：以**手机端体验**为主（最新主流浏览器），兼容 PC 端自适应。
*   **数据架构**：当前阶段使用 localStorage 本地存储。数据结构设计时预留 id 字段，方便未来对接后端。

## 4. 自动化文档系统管理
**以下文件通过 Hook 自动检测并提醒读取（见 `.claude/hooks/session-start.sh`）：**
*   **`Master Plan.md`**：需求文档、需求总揽。
*   **`Tasks.md`**：当前任务清单与下一步计划。
*   **`Rules.md`**：**项目长时记忆**。记录所有修好的 Bug 和教训。优先读取父目录共享版本。
*   **`DaMo-Design-System`**：视觉规范（颜色、布局、交互细节）。

**更新规则：**
*   完成任务后 → 更新 `Tasks.md`
*   修复 Bug 后 → 记录到 `Rules.md`（包含原因和解法）
*   需求变更后 → 更新 `Master Plan.md`

## 5. 上下文管理协议

@.claude/docs/context-management.md

**核心原则：文件是记忆，对话是临时的。**
*   重要决策必须写入文件，不依赖对话历史。
*   长会话中主动建议压缩或清理上下文。
*   压缩后按 `Tasks.md → 修改中的文件 → Rules.md` 的顺序恢复上下文。

## 6. 调试协作
*   若修复失败，请在代码中加入 `console.log`，由我反馈浏览器日志进行分析。
*   重大修改前提醒我进行 Git 提交或备份。

## 7. 协作流程

每次修改都严格遵循以下步骤：

1. **理解需求**：读取相关文档和代码，确认我要什么。
2. **提出方案**：用简单的中文描述修改计划（改哪些文件、怎么改、为什么这样改）。
3. **等我确认**：我说"好"或"可以"后才动手写代码。不确定的地方**必须问我**，不能自己猜。
4. **写代码**：按计划执行，每次只改 1-2 个文件。
5. **我来验收**：改完后告诉我怎么测试，由我在浏览器中确认效果。

**多文件修改时**：先列出所有受影响的文件清单，获得确认后再逐一修改。
